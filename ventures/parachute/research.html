Parachute Computer

Open-source AI memory infrastructure — knowledge graph, agent runtime, and composable module ecosystem for humans and machines.

April 22, 2026 Open Parachute, PBC · Colorado parachute.computer github.com/ParachuteComputer

What It Is

Parachute is building open infrastructure for AI memory — the layer between humans and AI systems that holds structured, persistent knowledge. The core thesis: conversation history is not memory. Notes, tags, links, and relationships in a traversable graph are memory. Any AI (Claude, ChatGPT, Gemini) gets access to that graph via the Model Context Protocol (MCP), without vendor lock-in.

The project is organized as Open Parachute, PBC — a Colorado Public Benefit Corporation. A structural sibling to RegenHub, LCA: both operate from Colorado, both mission-driven entities in the cooperative / public-benefit space.

Design stance: Local-first, AGPL-licensed, single-tenant-by-default. Data lives in SQLite on your machine. The hosted cloud offering wraps the same code. Third parties can build modules under any namespace — no central registry required.

Module Protocol — The Composability Layer

The most architecturally significant piece is the module protocol — a set of canonical HTTP contracts any service can implement to plug into the Parachute hub as a first-class citizen:

EndpointPurpose
GET /.parachute/infoModule identity — name, version, kind, capabilities. No auth, CORS *.
GET /.parachute/icon.svgInline SVG for hub display. ~200 bytes.
GET /.parachute/config/schemaJSON Schema of configurable properties (Phase 2+).
GET /.parachute/configCurrent config values (requires admin scope).
GET /.well-known/parachute.jsonService manifest — declared scopes, dependencies, endpoints, startCmd.

Modules declare dependencies and the hub auto-wires them. A module that needs Scribe declares it in its manifest; when both are installed, the hub injects Scribe's URL automatically. The ambition is a self-assembling ecosystem — add a module, capabilities appear without manual configuration.

Third-party modules don't need a central registry. Any npm package ships /.parachute/module.json and becomes a first-class participant.

Identity — Hub as OAuth Issuer

The hub is the identity authority. OAuth 2.1 + PKCE. Services are OAuth resource servers that validate hub-issued tokens — identity belongs at the front door, not embedded in a sibling data service.

Scope model

  • vault:read, vault:write — vault data access (write implies read)
  • scribe:transcribe — audio transcription
  • hub:admin — ecosystem configuration
  • channel:send — messaging
  • vault:<name>:read — multi-vault scopes (planned, Phase 2)

Service catalog in token

The OAuth token response carries a services object (vault URL, scribe URL, versions). Clients configure themselves from one auth exchange — no manual endpoint wiring required.

Rollout phases

  • Phase 0 (today): Hub proxies OAuth to Vault's own implementation
  • Phase 1: Token carries full service catalog
  • Phase 2: Real scope enforcement; step-up permissions
  • Phase 3: Auth extracted to dedicated module

Cloud Offering

Parachute Cloud wraps the self-hosted vault in hosted multi-tenant infrastructure. One Cloudflare Worker dispatcher routes by hostname to isolated Durable Objects (one per vault). R2 for attachments. D1 for account/subscription metadata. Custom Hostnames for wildcard SSL without Enterprise billing.

Strengths

MCP as lingua franca. Building on MCP is the correct bet. Any agent that speaks MCP gets vault access without model lock-in. The ecosystem doesn't depend on one AI provider surviving.

Module protocol is genuinely extensible. The /.parachute/* contract set is clean. Third-party modules don't need a central registry. An LCA-specific module — CIS integration, patronage accounting read layer — could be built as a standard Parachute module.

Cloudflare-native architecture. Durable Objects per vault is the right isolation model for multi-tenant. Avoids shared-state contention. Geographically distributed. Edge-native from day one.

AGPL + Colorado PBC. Values alignment with RegenHub, LCA. Open source as trust. Public benefit chartered. Both entities emerging in the same state, same era, adjacent to cooperative/solidarity economy circles.

Voice-native pipeline. Parachute Daily + Scribe + Narrate form a voice-first journaling loop. For an LCA capturing meeting knowledge: voice → transcription → structured vault note → agent-readable context is high-value workflow.

Where It's Early

Cloud offering is scaffold only. The architecture is well-designed but not deployed. No enterprise tier, no team vaults, no organizational pricing model.

Single-tenant by design, multi-tenant by exception. The core vault assumes one owner. Multi-user workflows require either separate vaults per person (with federation TBD) or upstream work on note-level ACL.

Agent framework is very early. The pattern is right — markdown-defined agents with vault integration. But it's pre-production. Stateful agents on Cloudflare Durable Objects are the right long-term shape; the runtime isn't there yet.

Relevance to Techne Studio / RegenHub, LCA

Immediate use cases for an LCA Parachute instance

  • Member knowledge base: Organizer notes, agreements, decisions, governance records — structured in a vault, queryable by Nou via MCP.
  • Meeting capture pipeline: Voice recording → Scribe → structured note → tagged in graph → Nou reads context for next meeting.
  • Agent context persistence: Nou and Dianoia's working memory across sessions. Currently flat workspace files; a vault gives structured, linkable, searchable context.
  • CIS bridge: Vault webhooks → Supabase CIS. Member records in Parachute, financial records in CIS, synchronized via agent-triggered hooks.
  • Patronage documentation: Each member's patronage log as vault notes with structured metadata. Agent-readable for audit and reporting.

Where LCA product research shapes enterprise Parachute

  • Multi-member vault model: Define what org-level vault looks like — member identity, role-based scopes, shared vs. private namespaces.
  • Structured note schemas: Agreements, capital account entries, governance votes — typed note schemas with required fields.
  • Agent access tiers: What scopes should an org's agent (Nou) have vs. a member's personal agent vs. a guest/external agent?
  • Webhook → external coordination: Parachute → Workshop surface. Meeting note created → sprint proposed automatically.
  • Cross-org federation: Two LCAs sharing selected vault content. Defines the federation model for Parachute in cooperative networks.

Technical Advisor Framing

The natural advisory focus is enterprise / organizational product research — using the LCA's operational needs as a living specification. This is valuable to Parachute because:

  • The LCA's governance, patronage accounting, and multi-agent coordination requirements are harder than typical enterprise SaaS.
  • Techne's work (CIS, Workshop, cooperation.games) produces concrete integration requirements — not hypothetical features, but real workflows needing vault support.
  • RegenHub's network connects Parachute to the cooperative/solidarity economy space — a distribution channel for values-aligned enterprise adoption.

Proposed research focus areas

  • Org-level vault architecture — multi-member, role-scoped, federation-ready
  • Structured note schemas and REA ontology compatibility
  • Agent access tiers for organizational deployments
  • Webhook integration patterns for external coordination surfaces
  • Pricing and packaging for cooperatives and small orgs (between Personal+ and Pro)
