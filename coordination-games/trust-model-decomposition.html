Trust Model Decomposition

Coordination Games Olympiad · Rolling back P496 · Rebuilding on Lucian's plugin architecture

P496 built a platform-authoritative trust taxonomy that conflicts with the plugin architecture documented in Lucian's trust-attestations.md. This document records what was built, why it doesn't fit, what's being removed, and what the rebuild foundation is.

P496 (commit 64c75f1) has been reverted in Roots-Trust-LCA/cooperation-games as of commit 3548688. The trust/ directory is back to the P487 visualization state.
01 — What P496 Built

A platform-authoritative trust taxonomy.

P496 delivered 14 EAS schema definitions, a 6-dimensional score composition model with a fixed weight matrix, and a public specification page. The architecture placed the platform in the position of deciding:

  • Which attestation types exist (14 schemas in 3 categories)
  • What each attestation type means (game-specific schemas: OathKept, OathBroken, etc.)
  • How attestations are weighted (composition.json — a 14×6 weight matrix)
  • What the six dimensions of trust are (cooperation, reliability, stewardship, coordination, influence, resilience)
  • How betrayal is weighted asymmetrically (1.5× hardcoded multiplier)
  • How attestations decay (time-based, per-schema half-lives)

The result was a coherent system with reasonable design choices — but the coherence was the problem. All interpretation was enshrined in the platform, which is exactly what Lucian's design philosophy rejects.

"Platform should not decide what is trustworthy. The platform provides signed statements and game event logs as raw data. All interpretation (scores, gating) lives in plugins."
— trust-attestations.md
02 — The Architectural Conflict

Two different theories of where authority should live.

P496 — Rolled back
Platform-Authoritative
Schema definition
14 fixed schemas defined by the platform. Attestation types are canonical.
Score computation
6-dimensional weight matrix baked into composition.json. Platform computes scores.
Betrayal asymmetry
1.5× hardcoded platform-wide. Not pluggable.
Decay model
Time-based, per-schema half-lives. Independent of activity level.
Interpretation
Platform defines bands (76–100 excellent, 0–25 poor). Meaning is prescribed.
Extensibility
Adding a new game requires a new platform schema. Platform bottleneck.
Plugin architecture — Rebuilding toward
Platform-Neutral
Schema definition
One attestation schema: polarity (+1/-1), size (0–1), scope (string), reason. No game-specific types.
Score computation
Plugin (@ctl/attestations) computes CONDUCT, SKILL:game, STEWARDSHIP. Platform stores raw data only.
Betrayal asymmetry
Not baked in. Emerges from attester weighting and market mechanics.
Decay model
Activity-based: older attestations are outweighed by newer ones about the same subject. Stewardship-scaled.
Interpretation
Plugin interprets. CONDUCT defaults to 0.5 (new players). Scoring is plugin-configurable.
Extensibility
New games use scope tags (skill:oathbreaker, skill:ctl). No platform change needed.
03 — The New Attestation Primitive

One schema. Binary polarity. Scope as a string.

The rebuild replaces 14 game-specific schemas with a single attestation structure. The platform stores raw signed statements. All interpretation — scoring, weighting, gating — is a plugin concern.

{
  attester: WalletAddress,
  subject: WalletAddress,
  polarity: +1 | -1,            // positive or negative, no neutral
  size: float [0,1],              // conviction strength, default 1.0
  scope: string,                 // see scope tags
  reason?: string,               // ≤ 1000 chars
  timestamp: int,
  signature: Signature           // EIP-712 per EAS off-chain spec
}

Scope tags define what an attestation is about. Three categories:

conduct — reserved
Platform integrity dimension. Rule-following, no cheating, interpersonal decency, no scamming or abandonment. This is the gating scope — public lobbies with min_conduct enforce on this. Has full market mechanics (P&L, stewardship).
skill:<game-id> — convention-based
Per-game skill opinions (skill:oathbreaker, skill:ctl, skill:shelling-point). Contributes to a score. No P&L for attester — skill has no ground truth to calibrate against, so stewardship doesn't track it.
anything else — free-form
Stored, queryable, not aggregated by the default plugin. Custom consumers and future plugins can interpret. This is how the system stays extensible without platform changes.
04 — The Plugin Model

All interpretation lives in @ctl/attestations.

The platform ships with one default plugin that runs server-side. Clients read results via API — they never run the algorithm. The plugin produces:

CONDUCT score
Per-subject platform integrity score. Full market mechanics — attesters open positions, realize P&L as the score moves, their stewardship tracks calibration quality. New players default to 0.5 (mild positive trust). Gated on by public lobbies with min_conduct.
SKILL:<game> score
Per-game skill opinion aggregation. No P&L for attesters (no ground truth for skill). New players default to 0 (neutral). Not used for lobby gating in the default plugin.
STEWARDSHIP score
Quality of a player's CONDUCT judgments. Single scalar per attester. Tracks how early and accurate their CONDUCT attestations were — earlier correct calls earn more. High stewardship amplifies an attester's voice on all scopes. Defaults to 0 for new players.

The key mechanic: attester weight = function of their CONDUCT and STEWARDSHIP. Low-CONDUCT attesters have almost no voice. High-stewardship attesters have amplified, longer-lived influence. This is the sybil resistance mechanism — not schema restrictions or platform-level rules.

05 — What We Keep and What We Drop

The P496 work is not wasted — it needs to move.

KEEP
The trust visualization (P487). The D3 force-directed graph at cooperation.games/trust/ is a presentation layer. It can be rebuilt to show CONDUCT scores, stewardship rankings, and attestation graphs once the plugin layer is implemented. The visualization concept is sound; it just needs to read from different data.
KEEP
Game event vocabulary as reference. The P496 schemas named real events (OathKept, OathBroken, FocalPointConverged, CommonsPolluted, etc.). These names are still useful — they describe what the game event logs contain. They become the raw signal that attesters reference in their reason field, not structured schemas the platform requires.
KEEP
Guardian NFT concept. P496's special schema for guardian recognition aligns with Lucian's Guardian NFT design — end-of-season recognition for top stewards, non-revocable, no functional power. The concept is sound; the implementation is in the plugin's seasonal mint, not an EAS schema.
DROP
14-schema taxonomy. Removed. The platform does not define per-game attestation types. Scope tags replace schemas. OathKept is now scope: "conduct" or scope: "skill:oathbreaker" per the attester's judgment, not a required schema field.
DROP
6-dimensional composition model. Removed. Cooperation, reliability, stewardship, coordination, influence, resilience were platform-defined dimensions with a fixed weight matrix. The plugin model produces CONDUCT + SKILL:game + STEWARDSHIP — a simpler, more defensible set that emerges from the market rather than editorial judgment.
DROP
Hardcoded betrayal multiplier (1.5×). Removed. Asymmetry in the new model emerges from the market — early correct negative attestations earn outsized P&L naturally, without a platform-baked multiplier.
DROP
trust/spec/index.html. Removed with P496 revert. Will be replaced by documentation of the plugin architecture once the rebuild is underway.
06 — Rebuild Foundation

What the next sprint builds toward.

The plugin architecture specification in trust-attestations.md is the canonical reference for the rebuild. It is currently marked "brainstorm — not implementation-ready." The trust visualization work and the enrollment/profile work described in P497–P498 should be reconsidered against this foundation before any implementation proceeds.

Key rebuild requirements from the spec:

Platform core: attestations table + signed endpoints
New attestations table: (id, attester_wallet, subject_wallet, polarity, size, scope, reason, timestamp, signature, closed_at). Signed by EIP-712. POST /api/attestations validates signature and slot capacity. Reads unauthenticated; writes require valid signature.
Daily Merkle anchor
Nightly cron posts Merkle root of that day's attestations to AttestationAnchor contract on OP Sepolia. ~$0.01/day. Provides third-party verifiability without per-attestation on-chain cost.
Plugin: CONDUCT auto-drip from game completions
Every cleanly-finished public game emits an internal +0.05 CONDUCT signal for each participant. Not stored as attestation — derived from game event logs. This is the scaffold for new players to build trust through clean play.
Activity-based decay, stewardship-scaled
Decay indexed to newer attestations about the same (subject, scope), not clock time. Trusted-attester voices persist longer than zero-stewardship voices — this is the sybil defense, replacing explicit rate limiting.
The enrollment and profile architecture (P497–P498) should be reconsidered against the plugin model before implementation. The key change: profile trust display should show CONDUCT score, SKILL:game scores, and attestation history — not a 6-dimensional radar chart derived from a fixed weight matrix.
